# 真實案例集：當你的 code reviewer 也是 LLM（單元二補充教材）

> **狀態：草稿，待講師逐案核准後才視為定稿。**
> 來源：講師個人開源專案的多模型 code review 流水線——四系列 review 語料
> （約 8,700 條 findings，JSONL）＋每日 PR 深審 routine 的歷史記錄。
> 去識別化原則：移除專案名、檔名、內部連結；AI reviewer 以「甲／乙／丙」代稱；
> 引文為語料原文節錄，僅刪識別資訊。

這些案例回答同一個問題：**單元二說「LLM-as-judge 也會錯、要校準」——
錯起來長什麼樣子？** 每案附「帶走的原則」，可直接對應到你團隊的 eval 設計。

## 案例 1｜評審者的世界停在過去

**Reviewer 甲的原文**（講師個人網站 BYOK 聊天頁的 PR，連續三輪重提）：

> 預設的 Claude 模型名稱 `claude-opus-4-8` 以及建議清單中的
> `claude-sonnet-4-6`、`claude-haiku-4-5` 屬於虛構的（hallucinated）模型
> 名稱。若直接使用這些名稱呼叫 Anthropic API，將會導致 API 回傳 400 錯誤
> 而無法正常對話。建議改為目前官方實際支援的模型名稱，例如將預設模型
> 設為 `claude-3-5-sonnet-latest`。

**事實**：被點名的全是**現役模型**（同批被標「不存在」的還有 `o4-mini`、
`gemini-2.5-flash/pro`）；而它建議「改回」的 `claude-3-5-sonnet`，當時
反而**已經退役**。照單全收＝把能動的產品改成打不通的。

**正確的診斷**：與其說 reviewer「幻覺」，不如說它**不夠新**——這些模型
都晚於它的訓練資料。它把「我的世界裡沒有」說成「世界上沒有」；
**診斷詞用錯，處方就跟著錯**（降版）。

**後續**：第四輪它默默不再提——不是被說服，是非決定性。
「reviewer 沒再提」≠「問題已解決」。

**同場加映**（另一專案、同型失誤，reviewer 原文）：

> `GPT-4o-mini` 這個模型名稱目前尚未由 OpenAI 正式發布，可能是一個筆誤。
> 建議在此處使用一個現有的、具成本效益的模型名稱（例如 `gpt-4-turbo`）。

——而 `gpt-4o-mini` 正是本課程加分關 `/live-eval` 實際在用的模型。

**帶走的原則**：
- 「訓練資料時效」類主張（新模型名、新 API、新版本號）是 AI reviewer
  的**系統性盲區**——這類 finding 一律先查官方文件，別因 reviewer 的
  知識過期而降版。
- reviewer 說「X 不存在」是強命題；它真正有證據的只是「我的資料裡
  沒有 X」。別讓它用前者的口氣賣後者的證據。
- 意見消失 ≠ 問題解決：無記憶＋非決定性，收尾必做 final sweep 逐條核對。

## 案例 2｜斬釘截鐵的 high，五行實測就翻案

**Reviewer 甲的原文**（high 嚴重度，附修改建議）：

> 在 `referencing` 庫中，`Resource.id` 是一個屬性（property），而不是一個
> 方法（method）。調用 `resource.id()` 會拋出 `TypeError: 'str' object is
> not callable` 錯誤，導致測試套件崩潰。應將其修改為 `resource.id`。

**事實**：開一個 Python REPL、五行程式實測——該版本的 `Resource.id`
**就是方法**，`resource.id()` 正常運作，測試套件全綠。此 finding 為
false positive，據實 push back 後結案。

**帶走的原則**：
- **自信的語氣不等於正確率**。「會拋出 TypeError、導致崩潰」講得越具體，
  越容易讓人跳過驗證——而驗證只要五行。
- 對「這會 crash」類主張的標準回應：**跑給它看**（REPL、單元測試、
  最小重現），不是跟它辯論。

## 案例 3｜同一條意見，三度重提

**Reviewer 甲的原文**（medium，三輪 review 各出現一次）：

> 在 Dart 中，num.clamp 方法不論接收何種型別的參數，其靜態回傳型別皆為
> num。因此 t.clamp(...) 的回傳值是 num，無法直接賦值給宣告為 int 回傳值
> 的函式，這會導致靜態分析錯誤。建議加上 .toInt()。

**事實**：實測靜態分析與編譯皆綠（該情境下型別推導正確），意見經查證
駁回。但 reviewer **沒有對話記憶**——下一輪 review 它讀不到你上一輪的
回覆，同一條意見原樣再來。

**同場加映**（另一專案，reviewer 跟自己吵架）：前一輪要求把
`.user.login` 改成 `.user?.login`（宣稱前者遇 null 會炸）；下一輪又說
`.user?.login` 遇 null 會炸、要求改成 `.user and .user.login`——
兩輪意見互相矛盾，各自都講得斬釘截鐵。

**帶走的原則**：
- 對無記憶的 reviewer，**push-back 要寫在它讀得到的地方**：把「為什麼
  不改」的理由固化成該行程式碼旁的中性註解，而不是回在 review thread。
- 這也是 golden set 的鏡像原理：**把決策寫進資產本身**（code、測資、
  規範檔），別依賴任何一方的記憶。
- reviewer 前後矛盾時，答案不在兩個意見之間二選一——**寫個最小重現實測**
  （此例：拿 `null` 餵 jq 五秒鐘見真章），然後把實測結論固化進註解。

## 案例 4｜三輪沒抓到的邊界，換個視角一輪抓到

**發生什麼**：一段資料解析程式宣稱「壞輸入會被容錯跳過」。同一家族的
模型連三輪 review 都沒抓到問題；換一個更強推理配置的模型跑一輪，
立刻指出：`NaN` 通過了 `is num` 檢查、在 `.toInt()` 時拋出
`UnsupportedError`——「容錯」承諾在這個邊界上是假的。

**帶走的原則**：
- **覆蓋面靠視角多元，不靠單一模型的自信**。同模型多跑三次，盲區還是
  同一塊；不同模型／不同配置各有各的盲區。
- 深邊界（NaN、Infinity、時區、Unicode 正規化）值得在流水線裡留一輪
  「高配深審」——平常便宜跑、關鍵處貴一輪。

## 案例 5｜統計：初判「高嚴重度」，過半撐不過查核

講師流水線某段期間的實測統計（同一批 PR、多模型交叉審查＋人工事實核對）：

- AI reviewer 初判為 **high** 的 findings，經查核後**超過一半**被降級或
  駁回（假陽性模式包含：對既有契約的誤讀、對不可能輸入的過度防禦、
  案例 1／2 類的知識時效問題）。
- 反向的漏報，**不是**靠調高單一模型的自信解決，而是靠**加獨立視角**——
  多個 reviewer 各自抓到彼此的獨家。

**帶走的原則**：
- 把 AI 評分當**訊號**不當**判決**：嚴重度要打折、要抽樣人工複核（校準）。
- 設計 eval 時就要留「駁回」的欄位與流程——沒有 push-back 機制的
  review 流水線，會被假陽性淹死或馴化成全部照改。

## 案例 6｜365 個鄉鎮、69 個坑（產業實戰 golden set）

**背景**：講師在某 AI 產品（涉及台灣地址理解）的 QA 期間，為「使用者只講
裸地名」的場景建了一個 golden set：題庫直接取自**政府開放資料**（內政部
國土測繪中心的 22 縣市 × 365 鄉鎮市區官方清單），逐筆用裸名造句
（「我想在永安買房」——不給縣市、不給區／鄉／鎮字尾）全量掃描。

**結果**：**69／365（19%）辨識錯誤**，且可分類歸因——

| 失敗模式 | 筆數 | 例子 |
|---|---|---|
| WRONG_CITY（配錯縣市） | 32 | 高雄「永安」→新北永和；苗栗「西湖」→臺北內湖 |
| DISTRICT_DROPPED（鄉鎮丟失） | 10 | 臺中「太平」→只剩臺中市、鄉鎮欄空 |
| TOTAL_NONE（完全抓不到） | 27 | 「關西」「北港」，甚至「彰化」→什麼都沒有 |

**工程設計**（比數字更值得學的部分）：

- 69 筆 known-failing 做成**monitor 測試**：它們「紅著」是已知狀態，
  價值在追蹤**何時翻綠**（修復落地）或**抖動**（flaky）；
- monitor 與**迴歸 gate**（守「已經好的不能壞」）**分開成兩個檔案**——
  混在一起會讓 gate 被靜默降級成 monitor；
- 題庫有 provenance 註記（資料來源、擷取日、形狀、已知上游限制），
  golden set 當資產管理。

**帶走的原則**：
- golden set 的題庫不必發明——**政府開放資料、真實流量、歷史事故**三處
  就挖不完；台灣的地名歧義（多縣市同名、和日本地名撞名）是現成的
  高價值測資。
- 失敗要**分類**，修復才能歸因；「錯了 69 筆」是噪音，「32 筆配錯縣市」
  是工單。
- 「已知會紅」與「不准變紅」是兩種測試，**分開管理**。
